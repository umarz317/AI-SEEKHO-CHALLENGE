1. Executive Development Direction
This plan refines the challenge solution into a practical execution brief for an autonomous development agent. The priority is a working agentic backend paired with a production-style mobile-first customer experience. The app must not look like a demo dashboard; it should feel like a real local-services booking app while still demonstrating planning, tool use, decision-making, state change, and follow-up automation.
The prototype should be reliable in mock mode, while remaining structurally ready for Google services such as Gemini, Google Places, Routes, Firestore, Firebase Cloud Messaging, and Cloud Functions.
Build goal
Parse a natural language service request in English, Urdu, or Roman Urdu.
Resolve service type, location, date, and time window.
Find candidate providers from mock data, with Google Places adapter compatibility.
Rank providers with deterministic, explainable scoring.
Create a simulated booking that writes real local state.
Schedule a simulated reminder and completion follow-up.
Produce trace logs and artifacts for judges and reviewers.
Non-goals for the development sprint
Do not build a desktop-style dashboard or expose technical traces in the main customer UI. Keep the experience mobile-first, polished enough to feel real, and focused on booking.
Do not build a payment flow.
Do not require real provider onboarding.
Do not depend on real WhatsApp, SMS, or push delivery for the demo.
Do not block the demo on API quotas, billing, API keys, or live internet availability.

15. Ranking Logic
The ranking logic should be deterministic and explainable. It must generate scores, reason codes, and user-facing explanations.
score =   0.30 * availabilityScore +   0.25 * ratingScore +   0.20 * distanceScore +   0.15 * reliabilityScore +   0.10 * responseScore  availabilityScore:   1.0 = available inside requested time window   0.6 = available same day but outside requested window   0.0 = unavailable  ratingScore = rating / 5  distanceScore = 1 - min(distanceKm / 10, 1)  reliabilityScore = min(completedJobs / 300, 1)  responseScore = responseRate

Reason code
Meaning
available_in_requested_window
Provider has a slot inside the user-requested time window.
closest_available_provider
Provider is the nearest provider that can serve in the requested window.
high_rating
Provider rating is strong relative to alternatives.
high_completed_jobs
Provider has strong job history.
fast_response_rate
Provider has a high simulated response rate.
outside_requested_time
Provider is available only outside the requested window.
too_far
Provider is farther than better options.
category_mismatch
Provider does not exactly match requested category.

Backend responsibility: The backend should return UI-ready explanations. The mobile app should display reasoning, not calculate it.
